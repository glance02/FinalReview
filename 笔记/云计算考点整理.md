## 1. 云计算概述

### 1.1 初识云计算

云计算是一种通过网络按需访问可配置计算资源池的模型。资源包括网络、服务器、存储、应用和服务等，可以快速提供和释放，并尽量减少管理成本和服务商干预。

通俗理解：

以前是“自己买服务器、自己部署、自己维护”；云计算是“像用水电一样使用 IT 资源”。用户不一定拥有硬件，只按需要购买计算、存储、平台或软件服务。

云计算和大数据关系：

- 云计算是处理大数据的重要手段。
- 大数据需要海量计算和存储；云计算提供弹性、分布式、可扩展的资源。
- 可以记成：大数据是问题规模，云计算是解决手段。

| ------- | ------------ | -------------------- |
| 公有云     | 云服务商面向公众提供服务 | 普通企业上云、互联网应用、阿里云、华为云 |
| 私有云/专属云 | 为一个组织单独建设和使用 | 政务、金融、保密要求高的企业       |

- 多个同类单位共同使用，如高校联盟、行业协会：社区云。
- 政务服务、医疗平台、教育平台等行业属性很强：行业云。

## 2. 分布式计算与分布式系统

### 2.1 分布式计算基本概念

定义：

分布式计算是把一个大任务拆分成多个小任务，分配到多个计算节点并行执行，最后汇总结果。

优点：

- 稀有资源可以共享。
- 可以在多台计算机上平衡计算负载。
- 可以把程序放在最适合运行它的计算机上。
- 适合处理单机难以承担的海量数据和计算任务。

一般步骤：

1. 设计分布式计算模型：规定组件如何运行、如何通信、如何管理。
2. 分布式任务分配：决定任务能否拆分、怎么拆分、分给哪些节点。
3. 编写并执行分布式程序：重点是任务划分和节点间通信。

### 2.2 分布式计算的理论基础

#### 2.2.1 ACID 原则[[02分布式计算.pdf#page=15&selection=0,6,0,10&color=yellow|p.15]]

ACID 是数据库事务正确执行的四个原则。

| 字母 | 含义                | 解释                           | 例子                             |
| ---- | ------------------- | ------------------------------ | -------------------------------- |
| A    | Atomicity，原子性   | 事务要么全成功，要么全失败回滚 | 转账扣款和入账必须同时成功       |
| C    | Consistency，一致性 | 事务前后都满足数据约束         | `a + b = 10` 事务后仍成立      |
| I    | Isolation，隔离性   | 并发事务互不干扰               | 转账未提交前，别人看不到中间状态 |
| D    | Durability，持久性  | 提交后的修改永久保存           | 宕机后已提交数据不丢             |

ACID 偏向强一致性，常见于传统关系数据库和金融交易场景。

#### 2.2.2 CAP 理论[[02分布式计算.pdf#page=21&selection=0,6,1,2&color=yellow|p.21]]

CAP 三要素：

| 字母 | 含义                            | 解释                                         |
| ---- | ------------------------------- | -------------------------------------------- |
| C    | Consistency，一致性             | 更新成功后，所有节点同一时间看到相同的新数据 |
| A    | Availability，可用性            | 每个非故障节点都能在正常时间内响应请求       |
| P    | Partition tolerance，分区容错性 | 网络分区或部分节点故障时，系统仍能继续运行   |

结论：

分布式系统无法同时完美满足 C、A、P。当网络分区P发生时，分布式系统无法同时保证强一致性C和完全可用性A。由于网络分区在分布式系统中不可避免，所以工程上通常在 CP 和 AP 之间取舍。

CAP 取舍[[02分布式计算.pdf#page=27&selection=2,0,3,2&color=yellow|p.27]]：

| 类型 | 保证              | 放弃/弱化 | 典型理解                               |
| ---- | ----------------- | --------- | -------------------------------------- |
| CA   | 一致性 + 可用性   | 分区容错  | 单机或不考虑网络分区的系统             |
| CP   | 一致性 + 分区容错 | 可用性    | 网络分区时宁可阻塞，也不返回旧数据     |
| AP   | 可用性 + 分区容错 | 强一致性  | 网络分区时先返回本地旧数据，之后再同步 |

场景题模板：

- 银行余额、订单支付：更偏 CP，因为不能乱。
- 社交动态、点赞数、DNS：更偏 AP，因为短时间不一致可以接受。
- 如果题目说“网络断开，N2 没同步到新数据”：
  返回旧数据 = 牺牲 C 保 A；阻塞等待 = 牺牲 A 保 C。

#### 2.2.3 BASE 理论[[02分布式计算.pdf#page=29&selection=0,6,1,2&color=yellow|p.29]]

BASE 是 CAP 的工程化延伸，核心思想是牺牲强一致性A来换取高可用，最后达到最终一致。BASE 是 AP 系统常见的工程哲学。

| 组成                             | 含义                                   | 例子                           |
| -------------------------------- | -------------------------------------- | ------------------------------ |
| Basically Available，基本可用    | 故障时允许损失部分可用性，保证核心功能 | 秒杀排队、限流、降级页面       |
| Soft State，软状态               | 允许系统存在中间不一致状态             | 多副本异步同步过程中短暂不一致 |
| Eventual Consistency，最终一致性 | 一段时间后所有副本最终一致             | DNS 更新后各地缓存逐步变新     |

ACID 和 BASE 对比：

| 对比项   | ACID                 | BASE                       |
| -------- | -------------------- | -------------------------- |
| 目标     | 强一致性、事务可靠   | 高可用、可扩展、最终一致   |
| 典型系统 | 关系数据库、交易系统 | 大规模分布式系统、NoSQL    |
| 思想     | 不允许中间错误暴露   | 允许短暂不一致，但最终收敛 |

ACID -> CAP -> BASE 的演进逻辑：

- ACID 解决的是传统数据库事务问题，重点是“数据一定正确”，适合单机或强一致事务场景。
- 到了分布式系统，节点之间要通过网络通信，网络分区、节点故障、延迟都可能发生，所以只讲 ACID 不够，需要 CAP 来说明一致性、可用性、分区容错性之间的取舍。
- CAP 告诉我们三者不能同时完美满足，但没有直接告诉工程上怎么做；BASE 就是在 CAP 基础上的工程策略：放弃强一致，保证基本可用，允许软状态，最后达到最终一致。
- 考试遇到场景题时，不要只背定义，要先判断业务更怕“数据错”还是更怕“服务不可用”。怕数据错偏 CP/ACID；怕服务不可用偏 AP/BASE。

#### 2.2.4 最终一致性[[02分布式计算.pdf#page=34&selection=1,1,1,6&color=yellow|p.34]]

| 类型       | 含义                                                        | 例子                       |
| ---------- | ----------------------------------------------------------- | -------------------------- |
| 强一致性   | 写成功后，后续所有读都能读到最新值，即CAP中的C              | 银行存款后立刻显示新余额   |
| 弱一致性   | 写成功后，不保证立刻读到最新值                              | 发动态后别人可能稍后才看到 |
| 最终一致性 | 没有新写入时，经过一段时间所有读最终读到最新值，即BASE中的E | DNS、异步复制系统          |

最终一致性变体：

- 因果一致性：有因果关系的进程必须读到相关更新。
- Read-your-writes：自己写入后，自己后续读取一定能读到。
- Session consistency：一个会话内保证 Read-your-writes。
- 单调读：读到新值后，不会再读到更旧的值。
- 单调写：同一进程的写操作按顺序执行。

#### 2.2.5 一致性散列[[02分布式计算.pdf#page=34&selection=1,1,1,6&color=yellow|p.34]]

为什么需要：

普通 `hash(key) % N` 在服务器数量 N 改变时，大量 key 会重新映射，导致大规模数据迁移。

一致性散列思想：

1. 把 hash 空间组织成一个虚拟圆环。
2. 服务器和数据对象都映射到这个环上。
3. 数据对象顺时针找到遇到的第一个服务器作为存储节点。
4. 增加或删除服务器时，只影响相邻区间的数据，而不是全局重分布。

用途：

- 分布式缓存。
- 分布式存储。
- 节点动态扩缩容。
- 减少服务器变动时的数据迁移量。

### 2.3 分布式系统概述[[02分布式计算.pdf#page=52&selection=2,0,2,7&color=yellow|p.52]]

分布式系统的关键特性：

- 容错性：某些节点故障时系统仍能工作。
- 高可扩展性：增加普通服务器即可提升整体能力。
- 开放性：组件接口开放，便于接入和扩展。
- 并发处理能力：多个节点并行执行任务。
- 透明性：用户不需要关心底层节点在哪里、如何通信。


#### 分布式存储

四种类型

| 类型                | 基本存储单位                 | 访问接口/协议                    | 主要特点                                                         | 适用场景                                     |
| ------------------- | ---------------------------- | -------------------------------- | ---------------------------------------------------------------- | -------------------------------------------- |
| 分布式块存储        | 数据块                       | SCSI、FC、SATA、SAN等            | 性能高，支持随机I/O，像磁盘一样给上层系统使用                    | 数据库、虚拟机磁盘、交易型和频繁变化的数据   |
| 分布式文件存储      | 文件                         | NFS、CIFS、POSIX                 | 提供统一文件系统视图，便于文件共享，可支持频繁修改和删除         | 图片、视频、邮件附件、文件共享、在线备份     |
| 分布式对象存储      | 对象，即带自定义元数据的文件 | HTTP REST、SOAP                  | 对象有唯一ID和元数据，扩展性强，适合一次写多次读                 | 云存储、备份归档、相对静态的海量非结构化数据 |
| 分布式表存储 / NoSQL | 表、文档、Key-Value、列族等  | Apache协议、HTTP/REST、数据库API | 面向结构化/半结构化数据，高扩展、高并发，通常弱化SQL/Join/强事务 | 社交网络、日志、微博、缓存、海量数据查询分析 |


区别：

- 抽象层次不同：块存储最接近物理磁盘，把空间切成固定大小的数据块；文件存储在块之上提供文件和目录；对象存储把数据封装成带唯一 ID 和元数据的对象；表存储直接面向应用数据模型，存储结构化或半结构化数据。
- 接口不同：块存储使用 SCSI、FC、SAN 等块设备接口；文件存储使用 NFS、CIFS、POSIX 等文件接口；对象存储多使用 REST/SOAP 接口；表存储使用数据库或 NoSQL 接口。
- 性能特点不同：块存储没有复杂打包/解包过程，随机 I/O 性能好；文件存储协议开销比块存储高，但共享和管理方便；对象存储协议开销更高，不适合频繁小修改，但扩展性强；表存储强调高并发、大数据量和灵活数据模型。
- 更新方式不同：块存储和文件存储通常支持就地更新；对象存储通常通过创建新对象版本更新，适合一次写多次读；表存储的更新方式取决于具体数据库，常用于大规模写入、查询或分析。
- 数据类型不同：块存储不关心数据语义，数据库、虚拟机磁盘等都能放；文件存储主要放图片、视频、文档、附件等非结构化文件；对象存储主要放海量、相对静态的非结构化数据；表存储主要放结构化/半结构化数据，如日志、用户状态、JSON 文档、列族数据等。

联系：

- 它们都属于云存储/分布式存储体系，目标都是把多台机器、多块磁盘组织成可扩展的存储资源池，对外提供统一服务。
- 它们都依赖分布式系统的共性技术，如副本、负载均衡、容错、元数据管理、一致性散列、CAP 取舍和最终一致性等。
- 它们可以分层组合：块存储可以作为文件系统或数据库底层；文件存储可以支撑上层数据处理；对象存储可以基于 HDFS 等分布式文件系统构建；表存储/NoSQL 也可能把底层数据落到分布式文件系统或对象存储中。
- 本质上，它们是在性能、共享能力、扩展性、管理复杂度和一致性之间做不同取舍：块存储偏性能，文件存储偏共享，对象存储偏海量扩展，表存储偏应用数据模型和大规模读写查询。


#### 2.3.3 分布式存储系统实例：Apache Hadoop

Hadoop 是 Apache 的开源分布式存储与计算框架，核心包括：

- HDFS：分布式文件系统，负责存储。
- MapReduce：分布式计算模型，负责计算。
- Hadoop 2.x 后加入 YARN：负责资源调度。

HDFS 特点：

- 高容错：数据块有多副本。
- 高可扩展：可以通过增加普通服务器横向扩容。
- 高吞吐：适合大文件顺序读写。
- 低成本：可以部署在廉价商用服务器上。

HDFS 不适合（读写限制前面加一个不适合即可）：

- 低延迟访问，如毫秒级随机读。
- 大量小文件，因为会占用 NameNode 内存。
- 多用户并发写同一个文件。
- 文件随机修改，HDFS 更适合 append 追加。

HDFS 角色[[02分布式计算.pdf#page=79&selection=2,0,3,7&color=yellow|p.79]]：

| 角色               | 作用                                                                           |
| ------------------ | ------------------------------------------------------------------------------ |
| NameNode           | 主节点，管理文件系统元数据、目录树、文件到块的映射、访问权限；不直接存真实数据 |
| DataNode           | 从节点，实际存储数据块，执行创建、删除、复制块等操作                           |
| Secondary NameNode | 辅助 NameNode 合并 Fsimage 和 edits log，加快恢复；不是实时热备替代品          |

重要机制：

- 主从架构：一个 NameNode + 多个 DataNode。
- 分块存储：文件切成 block，默认常见大小为 128MB。
- 副本机制：每个 block 多副本存储，常见副本数为 3。
- 元数据管理：NameNode 保存文件属性和 block 位置映射。
- 心跳机制：DataNode 周期性向 NameNode 发送 heartbeat，NameNode 通过心跳判断 DataNode 是否存活；如果长时间收不到心跳，就认为该节点故障，并安排副本恢复。
- 块汇报机制：DataNode 会向 NameNode 汇报自己保存了哪些 block，NameNode 由此维护“文件 -> block -> DataNode”的位置关系。

HDFS 文件写入流程：

1. Client 向 NameNode 请求创建文件。
2. NameNode 检查文件是否已存在、权限是否允许，并决定每个 block 应该写到哪些 DataNode。
3. Client 把文件切成 block，按 pipeline 管道写入多个 DataNode。
4. DataNode 之间按副本链路转发数据，写完后逐级返回确认。
5. Client 通知 NameNode 写入完成，NameNode 更新元数据。

HDFS 文件读取流程：

1. Client 向 NameNode 请求读取文件。
2. NameNode 返回该文件各 block 所在的 DataNode 列表。
3. Client 按就近原则直接从 DataNode 读取数据，实际数据不经过 NameNode。
4. 如果某个 DataNode 失败，Client 切换到保存同一 block 副本的其他 DataNode。

考试写 HDFS 时要抓住一句话：

NameNode 管元数据和调度，DataNode 存真实数据，Client 读写数据时主要直接和 DataNode 通信，NameNode 不负责搬运真实文件内容。

#### MapReduce 模型

定义：

MapReduce 既是 Hadoop 的计算模块，也是一个计算模型。用户把算法拆成 Map 和 Reduce 两个阶段。

基本流程：

1. Input：输入文件放在 HDFS。
2. Split：输入被切成多个 split，分配给不同 Map 任务。
3. Map：每个 Map 处理自己的 split，输出中间 K-V。
4. Shuffle/Sort：系统按 key 分区、拉取、排序、分组，把相同 key 的 value 放一起。
5. Reduce：每个 Reduce 处理一组 key，合并 value 得到最终结果。
6. Output：结果写回 HDFS。

WordCount 例子：

输入：

```text
hello cloud
hello hadoop
cloud computing
```

Split：

- Split 1：`hello cloud`
- Split 2：`hello hadoop`
- Split 3：`cloud computing`

Map 输出 K-V：

```text
Map1: (hello,1), (cloud,1)
Map2: (hello,1), (hadoop,1)
Map3: (cloud,1), (computing,1)
```

Shuffle/Sort 后：

```text
(cloud, [1,1])
(computing, [1])
(hadoop, [1])
(hello, [1,1])
```

Reduce 输出：

```text
(cloud, 2)
(computing, 1)
(hadoop, 1)
(hello, 2)
```

考试写法：

- 先写 split 怎么划分。
- 再写 map 输出的每个 K-V。
- 再写 shuffle 按 key 聚合后的结果。
- 最后写 reduce 的最终 K-V。

### 2.4 分布式系统的进阶（了解）

## 3. 云计算架构

### 3.1 云计算的本质（这里不用背，理解就好）

云计算的本质是 IT 即服务。用户不再把 IT 看成必须自己购买、部署、维护的资产，而是按需获取服务。

好处：

- 降低初始投入。
- 避免资源超配。
- 能快速扩缩容。
- 运维责任部分转移给服务商。
- 用户更专注自己的核心业务。

### 3.3 云栈和云体

#### 3.3.1 云栈[[02分布式计算.pdf#page=52&selection=2,0,2,7&color=yellow|p.52]]

定义：

云栈又叫云平台，是在云上建造的运行环境。它支持应用发布、运行、监控、调度、伸缩，并提供访问控制、权限管理等辅助服务。

理解：

云栈是“纵向分层”。越往上越远离物理硬件，抽象程度越高，用户越容易使用。

三层云栈：

| 层   | 又称     | 提供               |
| ---- | -------- | ------------------ |
| IaaS | 效用计算 | 基础设施资源       |
| PaaS | 弹性计算 | 应用开发和运行平台 |
| SaaS | 随需应用 | 可直接使用的软件   |

四层云栈：

- 把 IaaS 中的硬件和虚拟化资源拆开。
- 一层为硬件层，一层为虚拟资源层，再往上是平台层和应用层。
  ![[Pasted image 20260708151453.png]]

五层云栈：

1. 固件/硬件层。
2. 软件内核层。
3. 云基本资源层。
4. 云软件环境层。
5. 云应用程序层。

#### 3.3.2 云体[[03云计算架构.pdf#page=34&selection=1,0,1,2&color=yellow|p.34]]

定义：

云体是云计算的物质基础，是云计算用到的资源集合。它是构成云计算的软硬件环境，如网络、服务器、存储器、交换机等，通过网络连接在一起。

包括：

- 服务器。
- 存储设备。
- 网络设备。
- 交换机。
- 数据中心辅助设施，如电力、空调、机架、冷却系统。

一句话区分：

- 云栈：纵向的软件/服务层次。
- 云体：横向的资源集合和数据中心实体。

#### 3.3.3 一切皆为服务

云计算的本质可以总结为：IT 即服务。

无论是公有云、私有云、混合云，还是 IaaS/PaaS/SaaS，核心都是把资源、平台、软件或能力包装成服务提供给用户。

## 4. 虚拟化技术

### 4.1 虚拟化定义

虚拟化是对计算资源的抽象，不只等于虚拟机。它可以为硬件平台、计算机系统、存储设备、网络资源等创建逻辑版本。

目的：

- 摆脱物理资源限制。
- 提高资源利用率。
- 实现隔离。
- 方便统一管理和调度。
- 支持按需分配。

#### 虚拟化技术分类[[04虚拟化技术（2）.pdf#page=5&selection=2,0,2,6&color=yellow|p.5]]

课件列出 4 类：

| 类型           | 含义                                                 |
| -------------- | ---------------------------------------------------- |
| 服务器虚拟化   | 一台物理服务器上运行多个虚拟机                       |
| 网络虚拟化     | 用软件再现物理网络功能，如虚拟交换机、路由器、防火墙 |
| 桌面虚拟化     | 桌面环境集中托管，用户远程访问                       |
| 软件定义的存储 | 把磁盘、闪存等抽象成统一存储池                       |

### 4.2 服务器虚拟化

#### 服务器虚拟化基本概念

| 概念                    | 含义                           |
| ----------------------- | ------------------------------ |
| Host Machine，物理机    | 真实物理服务器                 |
| Virtual Machine，虚拟机 | 虚拟出来的服务器               |
| Guest OS，客户操作系统  | 虚拟机中运行的操作系统         |
| Host OS，主机操作系统   | 物理机上运行的操作系统         |
| VMM                     | 虚拟机监视器，提供硬件资源抽象 |
| Hypervisor              | 虚拟化平台，托管和管理虚拟机   |

服务器虚拟化特性：

- 多实例：一台物理机运行多个虚拟机。
- 隔离性：虚拟机之间互不影响，数据隔离。
- 封装性：虚拟机可表现为一个文件或逻辑实体，便于迁移、备份、复制。
- 高性能：虚拟化开销要在可接受范围内。

#### 裸金属架构与寄生架构[[04虚拟化技术（2）.pdf#page=15&selection=7,0,7,7&color=yellow|p.15]]

| 类型                                                                       | 结构                         | 优点   | 缺点                   | 例子                                |
| ------------------------------------------------------------------------ | -------------------------- | ---- | -------------------- | --------------------------------- |
| 裸金属架构                                                                    | 硬件上直接运行 Hypervisor，再运行虚拟机  | 性能较好 | 实现复杂                 | Xen、VMware ESX、Hyper-V            |
| 寄生架构                                                                     | 先安装 Host OS，再安装 VMM，再运行虚拟机 | 实现简单 | 资源管理要经过 Host OS，开销较大 | KVM、VirtualBox、VMware Workstation |

![[Pasted image 20260708153355.png]] 

注意：课件中 KVM 被归到寄生架构代表产品，但它本身是 Linux 内核中的虚拟化模块，实际理解时要记住它依赖 Linux 内核和 QEMU。
#### CPU 虚拟化

问题来源：

x86 CPU 有 Ring 0 到 Ring 3 权限级别。操作系统内核原本运行在 Ring 0，但虚拟化层也需要最高权限，所以客户操作系统的一些敏感指令不能直接作用于真实硬件，必须被 VMM/Hypervisor 接管或模拟。

三种方案：

| 方案      | 核心思想                       | 优点            | 缺点            | 例子                   |
| ------- | -------------------------- | ------------- | ------------- | -------------------- |
| 全虚拟化    | 动态二进制翻译，敏感指令陷入 VMM 后转换执行   | 不修改客户 OS，兼容性强 | 性能损耗较大，VMM 复杂 | 早期 VMware、Virtual PC |
| 半虚拟化    | 修改客户 OS，把敏感指令替换为 Hypercall | 开销低，性能较好      | 需要修改 OS，兼容性差  | Xen 早期典型             |
| 硬件辅助虚拟化 | CPU 提供虚拟化指令和模式             | 性能好，兼容性更好     | 依赖硬件支持        | Intel VT、AMD-V       |

#### 内存虚拟化

目标：

把物理内存统一管理，包装成多个虚拟机各自看到的“虚拟物理内存”。

三种地址：

| 地址           | 含义                             |
| -------------- | -------------------------------- |
| 逻辑地址       | 客户 OS 给应用程序提供的线性地址 |
| 虚拟机物理地址 | 虚拟机看到的伪物理地址           |
| 机器地址       | 真实硬件地址                     |

两种实现：

- 页表写入法：半虚拟化常用，典型代表 Xen。[[04虚拟化技术（2）.pdf#page=31&selection=2,1,2,6&color=yellow|p.31]]
- 影子页表法：全虚拟化常用，VMware Workstation、VMware ESX、KVM 都采用过。[[04虚拟化技术（2）.pdf#page=32&selection=1,1,1,6&color=yellow|p.32]]

#### I/O 虚拟化[[04虚拟化技术（2）.pdf#page=33&selection=0,0,1,3&color=yellow|p.33]]

| 方法     | 含义                                   | 优缺点                           |
| -------- | -------------------------------------- | -------------------------------- |
| 全虚拟化 | VMM 模拟网卡、磁盘等 I/O 设备          | 对客户 OS 透明，但设计复杂       |
| 半虚拟化 | 客户 OS 安装前端驱动，VMM 提供后端驱动 | 性能较好，但需要驱动/修改        |
| 直接划分 | 把物理 I/O 设备直接分给虚拟机          | 开销低，但共享困难，需要更多硬件 |

#### 网络/带宽虚拟化：

- 网络虚拟化把物理网络抽象成多个逻辑网络，让不同虚拟机、租户或业务看到彼此隔离的网络环境。
- 带宽虚拟化可以把一条物理链路或一个物理网卡的带宽按策略分配给多个虚拟网络/虚拟机，例如限速、优先级、QoS、虚拟交换机队列等。
- 典型实现包括虚拟交换机、VLAN/VXLAN、SDN 控制器、虚拟网卡、虚拟路由器、防火墙和负载均衡等。

#### “一虚多”和“多虚一”：

| 形式   | 含义                             | 例子                                                                                             |
| ------ | -------------------------------- | ------------------------------------------------------------------------------------------------ |
| 一虚多 | 一个物理资源被虚拟成多个逻辑资源 | 一个 CPU 分成多个 vCPU，一台服务器跑多个虚拟机，一张物理网卡分给多个虚拟网络                     |
| 多虚一 | 多个物理资源被抽象成一个统一整体 | 多台服务器组成一个资源池，多个数据中心对外表现为一个云平台，分布式存储对外表现为一个统一存储系统 |

答题角度：

- 微观上常见“一虚多”，用于提高单个物理资源利用率和隔离多个用户。
- 宏观上常见“多虚一”，用于把分散资源池化，对外提供统一服务。

### 4.3 商用虚拟机技术（了解）

代表技术包括 Xen、KVM、Hyper-V、VMware ESX/ESXi、VMware Workstation 等。老师最后一课主要强调 KVM 实验和基本理解。

### 4.4 新型硬件虚拟化（了解）

代表方向包括 GPU、FPGA、RDMA、NVM 等新型硬件虚拟化。复习时知道它们属于硬件资源虚拟化扩展即可。

### 4.5 实践：Xen 虚拟化技术（了解）

了解 Xen 是一种直接运行在硬件上的虚拟机管理程序，早期以半虚拟化方案著名。

### 4.6 实践：KVM 虚拟化技术

KVM 全称 Kernel-based Virtual Machine，即基于内核的虚拟机。

要点：

- 需要 x86 架构和硬件虚拟化支持，如 Intel VT 或 AMD-V。
- 需要 QEMU/qemu-kvm 作为上层控制和设备模拟。
- 可在不改 Linux/Windows 镜像的情况下运行多个虚拟机。
- 可用 VirtIO 提供准虚拟化 I/O，提高性能。

### 4.7 轻量级虚拟化

#### 容器与 Docker

容器和虚拟机区别：

| 对比            | 虚拟机              | 容器                           |
| --------------- | ------------------- | ------------------------------ |
| 虚拟化层级      | 虚拟硬件 + Guest OS | 操作系统级别                   |
| 是否包含完整 OS | 通常包含            | 不包含完整内核，共享宿主机内核 |
| 启动速度        | 较慢                | 快                             |
| 资源开销        | 较大                | 小                             |
| 隔离强度        | 强                  | 相对弱一些                     |
| 适合            | 多 OS、强隔离       | 微服务、快速部署、弹性伸缩     |

Docker 本质：

Docker 容器本质上是宿主机上的一个进程。

背后三个关键技术[[04虚拟化技术（2）.pdf#page=80&selection=2,0,2,9&color=yellow|p.80]]：

| 技术                    | 作用                                                     |
| ----------------------- | -------------------------------------------------------- |
| namespace               | 资源隔离，如主机名、进程号、网络、挂载点、用户等         |
| cgroups                 | 资源限制、优先级控制、审计、控制进程组                   |
| Copy-on-write，写时复制 | 镜像只读层共享，容器修改时复制到读写层，提高文件操作效率 |

namespace 六类隔离：

| Namespace | 隔离内容                   |
| --------- | -------------------------- |
| UTS       | 主机名和域名               |
| IPC       | 信号量、消息队列、共享内存 |
| PID       | 进程编号                   |
| Network   | 网络设备、网络栈、端口     |
| Mount     | 挂载点/文件系统            |
| User      | 用户和用户组               |

#### WebAssembly/Wasm 轻量级虚拟化补充

课件重点是 Docker；如果老师补充了 Wasm，可按下面理解：

| 技术             | 使用场景                                                                       |
| ---------------- | ------------------------------------------------------------------------------ |
| Docker 容器      | 微服务部署、持续交付、开发测试环境一致性、云原生应用                           |
| WebAssembly/Wasm | 浏览器或边缘侧运行轻量沙箱代码、插件化扩展、需要快速启动和强沙箱隔离的函数计算 |

一句话：

Docker 更像“打包应用运行环境”，Wasm 更像“可移植、安全沙箱中的轻量代码运行时”。

Docker 和 Wasm 的场景区别：

| 对比     | Docker                                 | Wasm                                             |
| -------- | -------------------------------------- | ------------------------------------------------ |
| 抽象对象 | 应用及其运行环境                       | 编译后的可移植字节码模块                         |
| 依赖     | 依赖宿主机内核和容器运行时             | 可运行在 Wasm Runtime、浏览器、边缘平台等        |
| 优势     | 生态成熟，适合服务部署和环境一致性     | 启动更快、沙箱更细、跨平台能力强                 |
| 更适合   | 后端服务、微服务、实验环境、云原生部署 | 边缘计算、函数计算、插件系统、浏览器侧高性能代码 |

## 5. 云存储技术

### 5.1 存储基础知识

#### DAS、NAS、SAN[[05云存储技术.pdf#page=23&selection=4,0,4,7&color=yellow|p.23]]

| 类型  | 全称                              | 数据类型 | 传输/接口         | 优点               | 缺点                 | 典型应用     |
| --- | ------------------------------- | ---- | ------------- | ---------------- | ------------------ | -------- |
| DAS | Direct Attached Storage，直接连接存储  | 块级   | SCSI、FC       | 简单、低成本、性能较好      | 扩展差、难管理、利用率不高      | 单机/小规模应用 |
| NAS | Network Attached Storage，网络附加存储 | 文件级  | IP 网络         | 文件共享方便、部署简单、易管理  | 带宽瓶颈、扩展有限、不适合某些数据库 | 文件服务器    |
| SAN | Storage Area Network，存储区域网络     | 块级   | FC-SAN/IP-SAN | 高性能、高扩展、高可用、支持集群 | 成本高、配置复杂、需要专人维护    | 数据库、大型存储 |

记忆：

- DAS：直接连，简单但难扩展。
- NAS：走 IP，提供文件共享。
- SAN：专用存储网络，块级高性能。

#### RAID[[05云存储技术.pdf#page=23&selection=4,0,4,7&color=yellow|p.23]]

RAID 是廉价冗余磁盘阵列，通过把多个磁盘组合成一个逻辑磁盘，提高性能、容量或可靠性。

基本术语：

| 术语        | 含义                             |
| ----------- | -------------------------------- |
| 分区/Extent | 磁盘上地址连续的存储块           |
| 分块/Strip  | 把分区切成大小相等、地址相邻的块 |
| 条带/Stripe | 不同磁盘上位置相关的 strip 集合  |
| 软 RAID     | 依赖 OS 和 CPU 完成 RAID 功能    |
| 硬 RAID     | 使用专门 RAID 控制器，性能更好   |

RAID 级别：

| 级别   | 核心思想             | 容量                 | 优点                                | 缺点                       |
| ------ | -------------------- | -------------------- | ----------------------------------- | -------------------------- |
| RAID0  | 条带化，无冗余       | `N * 单盘容量`     | 性能最高、实现简单                  | 无容错，一个盘坏数据丢     |
| RAID1  | 镜像                 | `(N/2) * 单盘容量` | 100% 冗余，读性能可提升             | 成本高，空间利用率 50%     |
| RAID3  | 专用校验盘，XOR 校验 | `(N-1) * 单盘容量` | 顺序读写较好                        | 校验盘瓶颈                 |
| RAID5  | 分布式校验，XOR 校验 | `(N-1) * 单盘容量` | 读性能好，容错，改善校验盘瓶颈      | 写性能受校验影响，重建复杂 |
| RAID6  | 双校验               | `(N-2) * 单盘容量` | 可容忍两块盘同时故障                | 写慢，成本更高             |
| RAID10 | RAID1 + RAID0        | `(N/2) * 单盘容量` | 读写都好，容错好                    | 贵，利用率 50%             |
| RAID50 | RAID5 + RAID0        | 视组数而定           | 比单 RAID5 容纳更多磁盘、读性能更好 | 实现复杂，同组两盘坏会失效 |

RAID3 和 RAID5 区别：

- RAID3 更适合顺序存取。
- RAID5 更适合随机存取。

#### 热备与快照

热备：

建立 RAID 时指定一个热备盘。平时不参与计算；当某块磁盘故障时，热备盘自动接替并重构数据。

类型：

- 全局热备盘：对整个磁盘阵列的所有 RAID 组生效。
- 局部热备盘：只对某个 RAID 组生效。

快照：

快照是某个时间点上逻辑卷的映像。逻辑上像完整拷贝，但通常只保存变化前的数据，因此能用较少空间支持恢复、备份、测试、数据分析。

## 6. 云计算网络

### 6.1 网络基础

网络模型：

| 层         | 作用                       | 设备/协议例子   |
| ---------- | -------------------------- | --------------- |
| 物理层     | 传输光/电/无线信号         | 网线、光纤、Hub |
| 数据链路层 | 帧传输、MAC 地址、差错校验 | 交换机、以太网  |
| 网络层     | IP 地址和路由选择          | 路由器、IP      |
| 传输层     | 主机间可靠/不可靠传输      | TCP、UDP        |
| 应用层     | 应用程序通信               | HTTP、FTP、SMTP |

封装与分用：

- 发送时，每层加自己的报头，称为封装。
- 接收时，每层剥掉对应报头，交给上层，称为分用。
- 传输层叫段，网络层叫数据报，链路层叫帧。

关键术语：

| 术语     | 含义                             |
| -------- | -------------------------------- |
| 数据平面 | 承载和转发真实数据流量           |
| 控制平面 | 决定数据如何转发                 |
| 管理平面 | 面向管理员的配置、监控、运维接口 |
| 网络功能 | 防火墙、负载均衡、网关等逻辑功能 |

#### 6.1.2 覆盖网络 Overlay Network[[06云计算网络（3）.pdf#page=30&selection=2,0,2,4&color=yellow|p.30]]

定义：

覆盖网络是在已有底层网络之上构建逻辑连接和管理能力的技术。

理解：

底层物理网络不用大改，通过封装/隧道技术，在上面再“叠”出一个逻辑网络。

常用协议：

- GRE。
- VLAN。
- VXLAN。
- NVGRE。
- IPsec。

什么时候用覆盖网络：

- 底层物理网络已经存在，不想大规模改造硬件和拓扑。
- 需要在不同物理位置、不同三层网络之上构建统一逻辑网络。
- 需要多租户隔离、虚拟机迁移、跨机房通信等云计算场景。
- 典型做法是“封装 + 隧道”：原始报文外面再套一层协议头，在底层网络中传输。

#### GRE

GRE 全称 Generic Routing Encapsulation，通用路由封装协议。

作用：

在一种网络层协议上封装另一种协议，让被封装的数据报能够在另一种网络中传输。

过程：

1. 入口路由器收到原始报文。
2. 判断报文需要走 Tunnel。
3. 在原始报文外加 GRE 头和外层 IP 头。
4. 在底层网络中转发封装后的报文。
5. 出口路由器解封装，去掉外层 IP 头和 GRE 头。
6. 交给原协议继续处理。

缺点：

封装/解封装会增加额外开销，降低有效数据传输效率。

GRE、VLAN、VXLAN 对比：

| 技术  | 所在思路          | 主要作用                                   | 适用场景                           | 局限                                                 |
| ----- | ----------------- | ------------------------------------------ | ---------------------------------- | ---------------------------------------------------- |
| GRE   | 三层隧道封装      | 把一种网络层协议封装进另一种网络层协议传输 | 点到点隧道、VPN、跨网络互通        | 隧道能力强，但本身不专门解决大规模二层多租户标识问题 |
| VLAN  | 二层广播域划分    | 在交换网络中隔离广播域                     | 企业局域网、部门隔离、中小规模网络 | VLAN ID 数量有限，跨三层扩展困难                     |
| VXLAN | 覆盖网络/隧道封装 | 在三层网络上扩展二层网络，并支持大量租户   | 云数据中心、虚拟机迁移、多租户隔离 | 有封装开销，需要 VTEP 支持                           |

#### VLAN

VLAN 是 Virtual LAN，虚拟局域网。

作用：

在二层交换网络中划分广播域。同一 VLAN 内像同一局域网一样通信；不同 VLAN 默认不能直接通信。

为什么需要 VLAN：

- 减少广播风暴。
- 提高安全性。
- 让不同部门/租户逻辑隔离。

VLAN 间通信：

需要路由器或三层交换机，称为 VLAN 间路由。

分类：

| 类型      | 划分依据          | 优缺点                           |
| --------- | ----------------- | -------------------------------- |
| 静态 VLAN | 端口              | 简单清楚，但大量端口维护麻烦     |
| 动态 VLAN | MAC、子网、用户等 | 适合设备移动，但配置和管理更复杂 |

限制：

传统 VLAN ID 数量最多约 4094，在大规模云数据中心多租户环境中不够用。

#### VXLAN[[06云计算网络（3）.pdf#page=44&selection=2,0,3,1&color=red|p.44]]

VXLAN 全称 Virtual eXtensible LAN，虚拟扩展局域网。

定义：

VXLAN 是一种隧道封装技术，把二层以太网帧封装进四层 UDP 报文，再通过三层网络传输。

它解决的问题：

- VLAN ID 数量不够。
- 虚拟机数量和租户数量巨大。
- 需要跨三层网络扩展二层网络。
- 需要支持虚拟机迁移后 IP/VLAN 不变。

关键概念：

| 概念 | 含义                                                   |
| ---- | ------------------------------------------------------ |
| VTEP | VXLAN Tunnel Endpoint，VXLAN 隧道端点，负责封装/解封装 |
| VNI  | VXLAN Network Identifier，VXLAN 网络标识，类似 VLAN ID |
| BD   | Bridge Domain，大二层域，常与 VNI 对应                 |

传输过程：

1. 源主机发出原始二层以太网帧。
2. 源端 VTEP 判断需要进入 VXLAN 隧道。
3. VTEP 加 VXLAN 头、UDP 头、外层 IP 头。
4. 报文通过三层网络传输到目的 VTEP。
5. 目的 VTEP 解封装，还原原始二层帧。
6. 目的 VTEP 转发给目标主机。

VXLAN 和 VLAN 对比：

| 对比         | VLAN         | VXLAN                    |
| ------------ | ------------ | ------------------------ |
| 网络标识数量 | 约 4094      | VNI 24 位，数量大得多    |
| 扩展范围     | 二层网络内   | 可跨三层网络             |
| 适用         | 中小规模网络 | 大规模云数据中心、多租户 |
| 开销         | 小           | 封装/解封装开销更大      |

#### NVGRE 与 IPsec

NVGRE：

- 使用 GRE 封装实现网络虚拟化。
- 通过 24 位 TNI 标识租户网络。
- 目的也是解决 VLAN 数量有限和多租户隔离问题。
- 与 VXLAN 是竞争/类似方案。

IPsec：

- 为 IP 网络提供安全性的协议集合。
- 通过 IPsec 隧道加密传输数据。
- 适合在公网上连接多个私有网络。

IPsec 关键协议：

| 协议 | 作用                               |
| ---- | ---------------------------------- |
| IKE  | 协商安全联盟 SA 和密钥             |
| AH   | 提供数据源认证和完整性校验，不加密 |
| ESP  | 提供认证、完整性校验和加密         |

#### 6.1.3 大二层网络[[06云计算网络（3）.pdf#page=61&selection=2,0,2,5&color=yellow|p.61]]

定义：

大二层网络是在跨数据中心或跨区域的基础网络上，构建允许二层协议通信的覆盖网络。

为什么需要：

- 大型云服务商有多个数据中心。
- 需要虚拟机跨机房迁移。
- 希望迁移前后 IP/MAC/VLAN 不变。
- 希望支持二层下的多路径路由和负载均衡。

覆盖网络和大二层网络的区别：

| 对比     | 覆盖网络                                    | 大二层网络                                       |
| -------- | ------------------------------------------- | ------------------------------------------------ |
| 关注点   | 一种通用技术方法，在已有网络上构建逻辑网络  | 一种目标网络形态，让分散主机像在同一个二层网络里 |
| 实现方式 | 通常靠隧道封装，如 GRE、VXLAN、NVGRE、IPsec | 常常用覆盖网络实现，例如 VXLAN 扩展二层          |
| 主要目的 | 抽象底层网络、隔离租户、跨网络连接          | 支持二层互通、虚拟机迁移、IP/MAC 不变            |
| 使用场景 | 多租户云网络、VPN、跨机房逻辑网络           | 数据中心互联、云主机迁移、大规模二层域           |

一句话：

覆盖网络是“怎么做”的技术手段；大二层是“要达到什么效果”的网络形态。大二层通常可以用覆盖网络来实现。

#### 6.1.4 租户网络

租户：

接受云服务商直接服务的客户。

租户网络：

租户租用虚拟机、云盘等资源后，还需要虚拟网络来连接和管理自己的虚拟设备，这个虚拟化网络就是租户网络。

单租户和多租户：

| 架构   | 含义                             |
| ------ | -------------------------------- |
| 单租户 | 一套系统只服务一个租户           |
| 多租户 | 同一套云管理系统同时服务多个租户 |

多租户网络要求：

- 流量隔离。
- IP/MAC 可独立规划。
- 资源按需租用。
- 网络规模可扩展。

### 6.2 数据中心网络：云计算的骨架

传统三层架构：

| 层                 | 作用                                             |
| ------------------ | ------------------------------------------------ |
| 接入层 Access      | 服务器接入、VLAN 标记、二层转发，常见 ToR 交换机 |
| 汇聚层 Aggregation | 连接接入交换机，提供安全、QoS、分析等            |
| 核心层 Core        | 为进出数据中心流量提供高速转发                   |

南北流量和东西流量：

| 流量     | 含义                               |
| -------- | ---------------------------------- |
| 南北流量 | 客户端和数据中心内部服务之间的流量 |
| 东西流量 | 数据中心内部服务器之间的流量       |

现代数据中心中，东西流量常常远超南北流量。

CLOS/Fat-Tree：

- CLOS 思想：用多个小规模、低成本交换单元构建大规模网络。
- Fat-Tree 是数据中心常用的 CLOS 型网络架构。
- 目标是构建无阻塞或低阻塞网络。
- 好处是可以用大量廉价交换机构建高扩展网络。

### 6.3 网络虚拟化

定义：

网络虚拟化把网络层的一些功能从硬件中剥离出来，建立虚拟网络层，实现网络服务与物理网络解耦。

核心要求：

- 隔离性：不同租户流量不能互访。
- 可移动性：虚拟机可跨二层/三层迁移，IP/VLAN 不变。
- 可扩展性：逻辑网络数量和规模可扩展。

#### 6.3.1 灵活控制：软件定义网络 SDN[[06云计算网络（3）.pdf#page=85&selection=2,0,2,11&color=yellow|p.85]]

SDN 全称 Software-Defined Networking，软件定义网络。

软件定义网络（SDN，Software Defined Networking）是一种新型网络架构，它将传统网络设备中的控制功能和数据转发功能分离出来，通过集中式的控制器对网络进行统一管理和编程控制，从而让网络更加灵活、可配置、易管理。

核心思想：

- 控制平面和数据平面分离。
- 控制逻辑集中化。
- 网络可编程。
- 底层网络设施对上层应用透明。

SDN 架构：

| 层/接口    | 作用                                               |
| ---------- | -------------------------------------------------- |
| 应用层     | 提供网络业务应用，如拓扑可视化、配置管理、安全策略 |
| 控制层     | 维护网络状态、拓扑、设备信息，决定转发策略         |
| 基础设施层 | 交换机、路由器等转发设备，负责实际数据转发         |
| 北向接口   | 应用层调用控制层能力                               |
| 南向接口   | 控制层下发规则给基础设施层，屏蔽物理设备差异       |
| ![[Pasted image 20260708164659.png]]           |                                                    |

SDN 工作原理：

1. 数据包到达交换机。
2. 如果交换机已有规则，按规则转发。
3. 如果没有规则，向控制器请求。
4. 控制器根据全局网络状态计算策略。
5. 控制器下发流表。
6. 交换机按流表处理后续数据包。

OpenFlow：

OpenFlow 是 SDN 中常见的南向接口协议，用流表描述交换机如何匹配和处理数据包。

#### 6.3.2 快速部署：网络功能虚拟化 NFV[[06云计算网络（3）.pdf#page=113&selection=0,5,4,5&color=yellow|p.113]]

NFV 全称 Network Function Virtualization，网络功能虚拟化。

定义：

用虚拟化技术把传统专用网络硬件设备的功能封装成软件模块，运行在通用硬件上。

例子：

- 防火墙。
- 负载均衡器。
- 路由器。
- 深度包检测 DPI。
- VPN 网关。

SDN 和 NFV 区别：

| 对比     | SDN                         | NFV                              |
| -------- | --------------------------- | -------------------------------- |
| 解决问题 | 怎么灵活控制网络转发        | 怎么把网络功能从专用硬件变成软件 |
| 核心     | 控制/数据平面分离、集中控制 | 网络功能软件化、虚拟化           |
| 例子     | 控制器下发流表              | 虚拟防火墙、虚拟负载均衡         |

服务功能链 SFC：

把多个网络功能按顺序串起来，例如：流量先过防火墙，再过负载均衡，再到应用服务器。

## 附录 B. 高频简答题模板

### B.1 云计算定义是什么？

云计算是一种通过网络按需访问共享可配置计算资源池的模型，资源包括服务器、存储、网络、应用和服务等。它可以快速提供和释放资源，减少管理成本和服务商人工干预。通俗地说，就是把 IT 资源像水电一样作为服务使用。

### B.3 云栈和云体怎么理解？

云栈是云计算的纵向分层结构，从硬件层到资源层、平台层、应用层逐层抽象，典型表现是 IaaS、PaaS、SaaS。云体是云计算的横向资源集合，是云计算的物质基础，包括服务器、存储、网络、数据中心及其辅助设施。简单说，云栈是“服务层次”，云体是“资源实体”。

### B.4 CAP 理论怎么答？

CAP 指一致性 C、可用性 A、分区容错性 P。分布式系统无法同时完美满足三者。由于网络分区不可避免，工程上通常在一致性和可用性之间取舍。CP 系统在分区时宁可阻塞也要保证新数据一致；AP 系统在分区时优先响应请求，允许短暂返回旧数据，之后通过同步达到最终一致。

### B.5 BASE 和 ACID 区别？

ACID 强调事务的原子性、一致性、隔离性和持久性，追求强一致，适合关系数据库和金融交易。BASE 强调基本可用、软状态和最终一致，允许短暂不一致来换取高可用和高扩展，适合大规模分布式系统。

### B.6 全虚拟化和半虚拟化区别？

全虚拟化不修改客户操作系统，通过动态二进制翻译或陷入模拟处理敏感指令，兼容性好但性能损耗较大。半虚拟化需要修改客户操作系统，把敏感指令改成 Hypercall 主动调用虚拟化平台，性能开销较低，但兼容性和可移植性较差。

### B.7 Docker 为什么轻量？

Docker 容器不是完整虚拟机，而是宿主机上的进程。它共享宿主机内核，通过 namespace 做资源隔离，通过 cgroups 做资源限制，通过写时复制提高镜像和文件系统效率。因此启动快、占用小，适合微服务和云原生部署。

### B.9 VXLAN 解决什么问题？

VXLAN 是 VLAN 的扩展，通过把二层以太网帧封装进 UDP 报文，在三层网络上传输二层流量。它解决传统 VLAN ID 数量不足、云数据中心多租户隔离、跨三层网络扩展二层网络和虚拟机迁移的问题。关键组件是 VTEP，关键标识是 VNI。

### B.10 SDN 和 NFV 区别？

SDN 重点是控制网络，核心是控制平面和数据平面分离，通过集中控制器和可编程接口灵活控制转发。NFV 重点是实现网络功能，把防火墙、负载均衡等专用硬件功能软件化，运行在通用服务器上。SDN 解决“怎么控制网络”，NFV 解决“网络功能怎么部署”。

### B.11 HDFS 如何管理文件？

HDFS 采用主从架构。NameNode 负责管理文件系统元数据，包括目录树、文件属性、文件到 block 的映射、block 到 DataNode 的位置映射；DataNode 负责真实数据块的存储。文件写入时，Client 先向 NameNode 申请创建文件，NameNode 分配 DataNode，Client 再把数据按 block 写入多个 DataNode；文件读取时，Client 先问 NameNode 获取 block 位置，再直接从 DataNode 读取。DataNode 通过心跳向 NameNode 报告存活状态，通过块汇报告诉 NameNode 自己保存了哪些 block。

### B.12 覆盖网和大二层网络区别？

覆盖网络是在已有物理网络之上构建逻辑网络的技术手段，通常通过隧道封装实现，例如 GRE、VXLAN、NVGRE。大二层网络是一种网络效果或目标形态，它希望跨机房、跨三层网络的主机仍像在同一个二层网络中通信，常用于虚拟机迁移和数据中心互联。简单说，覆盖网络是实现方法，大二层是目标效果；大二层常常用 VXLAN 这类覆盖网络实现。
