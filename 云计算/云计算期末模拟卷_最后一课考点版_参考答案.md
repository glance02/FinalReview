## 一、云计算部署模式与服务模式分析（10 分）

1. 适合采用混合云。学校官网、招生宣传页面面向公网，适合放在公有云或公有云资源上；教务系统、成绩、财务等敏感数据访问受限，适合私有云或专属云；选课高峰临时扩容可以利用公有云弹性资源。因此整体更符合“核心数据在私有云，弹性和公开业务使用公有云”的混合云模式。
2. 在线办公软件属于 SaaS，用户直接使用软件，主要负责业务数据和账号权限，不负责底层服务器和平台维护。租用云主机属于 IaaS，云厂商提供计算、存储、网络等基础资源，用户负责操作系统、数据库、中间件和应用部署。开发运行平台属于 PaaS，平台提供运行环境、开发框架、数据库/中间件等能力，开发者主要负责应用代码和业务逻辑。
3. 多所高校共同建设教育资源共享平台，更接近社区云。因为社区云服务于具有共同需求的一组组织，这里共同需求是高校教育资源共享、统一管理和协作使用。

## 二、CAP、BASE 与一致性场景分析（10 分）

1. 网络故障对应 CAP 中的 P，即 Partition tolerance，分区容错性。发生网络分区时，A 机房的新订单状态无法及时同步到 B、C。如果 B、C 仍然响应查询，可能返回旧数据，破坏一致性；如果 B、C 拒绝响应以等待同步，又破坏可用性。因此在 P 存在时，系统不能同时完美保证 C 和 A。
2. “必须返回最新订单状态，否则拒绝服务”更接近 CP。它保证一致性和分区容错性，在网络分区时宁可不可用或部分不可用，也不返回旧数据，因此牺牲的是可用性 A。
3. “先返回旧状态或处理中状态，稍后自动同步”更接近 AP。它优先保证可用性和分区容错性，弱化强一致性。体现 BASE 中的 Basically Available（基本可用）、Soft State（允许中间状态）和 Eventual Consistency（最终一致性）。
4. 订单支付扣款更适合强一致，因为涉及资金和余额，不能出现重复扣款、少扣款或状态错误。商品评论数量统计更适合最终一致，因为短时间内数量略有延迟通常可以接受，系统更看重高并发和可用性。

## 三、一致性散列计算与扩容分析（5 分）

1. 扩容前服务器位置为 S1=10、S2=40、S3=75。对象顺时针找第一个服务器：K1=8 到 S1；K2=12 到 S2；K3=38 到 S2；K4=41 到 S3；K5=80 绕回到 S1。

| 对象 | 哈希值 | 扩容前节点 |
| --- | --- | --- |
| K1 | 8 | S1 |
| K2 | 12 | S2 |
| K3 | 38 | S2 |
| K4 | 41 | S3 |
| K5 | 80 | S1 |

2. 新增 S4=60 后，环上顺序为 S1=10、S2=40、S4=60、S3=75。K1=8 到 S1；K2=12 到 S2；K3=38 到 S2；K4=41 到 S4；K5=80 到 S1。

| 对象  | 哈希值 | 扩容后节点 |
| --- | --- | ----- |
| K1  | 8   | S1    |
| K2  | 12  | S2    |
| K3  | 38  | S2    |
| K4  | 41  | S4    |
| K5  | 80  | S1    |

3. 一致性散列新增节点时，只影响新节点在哈希环上接管的相邻区间。本题中 S4 只接管 `(40,60]` 区间的数据，因此只有 K4 从 S3 迁移到 S4，而不是所有 key 都重新取模分配。

## 四、虚拟化技术与轻量级虚拟化（10 分）

1. 全虚拟化通过 Hypervisor 模拟完整硬件环境，客户操作系统不需要修改，兼容性好，但敏感指令模拟或二进制翻译会带来较大性能开销。半虚拟化需要修改客户操作系统，让它通过 Hypercall 主动调用 Hypervisor，性能开销较低，但兼容性较差。硬件辅助虚拟化依赖 CPU 等硬件提供的虚拟化扩展，让敏感指令处理由硬件支持，兼顾兼容性和性能，是现代虚拟化常用方式。
2. Hypervisor 负责在物理硬件和虚拟机之间进行资源抽象、隔离、调度和管理，使多台虚拟机共享 CPU、内存、I/O、网络等资源。裸金属架构中 Hypervisor 直接运行在硬件上，性能和隔离性较好；寄生架构中 Hypervisor 运行在宿主操作系统之上，部署方便，但多了一层宿主系统开销。
3. Docker 轻量是因为容器不是完整虚拟机，而是宿主机上的隔离进程。它共享宿主机内核，通过 namespace 实现进程、网络、文件系统等隔离，通过 cgroups 限制 CPU、内存等资源，通过镜像分层和写时复制减少存储开销，因此启动快、占用资源少。
4. WebAssembly 适合边缘计算、浏览器端执行、插件沙箱、Serverless 函数、跨平台轻量运行等场景。Docker 通常在操作系统层运行容器，隔离的是进程和运行环境；WebAssembly 更接近可移植字节码运行时，通过沙箱执行编译后的模块，启动更轻，运行位置可以是浏览器、边缘节点或服务器运行时。

## 五、云计算网络：覆盖网、大二层、VXLAN 与 GRE（10 分）

1. 覆盖网络是在已有底层物理网络或三层 IP 网络之上，通过封装、隧道等方式构建出的逻辑网络。该场景底层网络已经存在，要求多租户隔离、跨数据中心迁移和较低改造成本，因此适合用覆盖网络在现有网络上叠加逻辑租户网络。
2. 大二层网络是让跨机房、跨三层网络的主机在逻辑上像处于同一个二层网络中，支持二层互通、IP/MAC 不变和虚拟机迁移。覆盖网络是实现手段，大二层是目标效果；大二层通常可以通过 VXLAN 等覆盖网络实现。
3. VLAN 使用 12 位 VLAN ID，数量约 4094，主要用于二层交换网络内的广播域划分，适合企业局域网和中小规模隔离。VXLAN 使用 24 位 VNI，标识空间远大于 VLAN，可以把二层以太网帧封装进 UDP 报文并跨三层网络传输，适合大规模云数据中心、多租户网络和虚拟机迁移场景。
4. GRE 的基本思想是在一种网络层协议外再封装 GRE 头和外层 IP 头，使报文通过隧道穿越另一种网络。GRE 更偏通用三层隧道和协议封装；VXLAN 更面向云数据中心二层扩展和大规模多租户隔离，重点是 VNI 标识和 VTEP 封装/解封装。
5. VTEP 是 VXLAN Tunnel Endpoint，负责 VXLAN 报文的封装和解封装。VNI 是 VXLAN Network Identifier，用来标识不同的逻辑二层网络或租户网络。

## 六、SDN 与 NFV 架构分析（10 分）

1. SDN 即软件定义网络，是把传统网络设备中的控制功能和数据转发功能分离，通过集中式控制器统一管理和编程控制网络的架构。控制平面负责计算路径、维护拓扑、生成策略；数据平面负责按照流表或转发表转发数据包。
2. SDN 基本架构可以分为三层和两个接口。应用层运行网络业务应用，如安全策略、流量调度、拓扑管理；控制层由 SDN 控制器组成，掌握全局网络状态并生成转发策略；基础设施层包括交换机、路由器等转发设备，负责实际转发。北向接口连接应用层和控制层，使上层应用调用网络能力；南向接口连接控制层和基础设施层，控制器通过它向设备下发流表和策略，典型协议是 OpenFlow。
3. NFV 即网络功能虚拟化，是把传统依赖专用硬件的网络功能软件化，使其运行在通用服务器、虚拟机或容器上。可虚拟化的功能包括防火墙、负载均衡器、VPN 网关、路由器、DPI、NAT 等。
4. SDN 解决“网络如何集中控制和灵活转发”的问题，核心是控制/数据平面分离和网络可编程；NFV 解决“网络功能如何从专用硬件变成可快速部署的软件”的问题。二者配合时，NFV 提供虚拟防火墙、负载均衡等功能实例，SDN 负责把流量按策略引导到这些功能实例，形成可编排的云网络和服务功能链。

## 七、HDFS 文件管理与故障恢复（20 分）

1. `log.dat` 大小为 300 MB，块大小为 128 MB，因此切分为 3 个 Block。B1 为 128 MB，B2 为 128 MB，B3 为 `300 - 128 - 128 = 44 MB`。
2. NameNode 是主节点，负责文件系统命名空间、目录树、权限、文件到 Block 的映射、Block 到 DataNode 的位置映射等元数据管理，不直接存储和传输真实数据。DataNode 是从节点，负责真实 Block 数据的存储、读取、写入、删除和副本复制，并向 NameNode 发送心跳和块汇报。Secondary NameNode 辅助 NameNode 合并 Fsimage 和 edits log，减小恢复成本，但不是 NameNode 的实时热备。
3. 写入流程：Client 先向 NameNode 请求创建 `log.dat`；NameNode 检查文件是否存在、权限是否允许，并为各 Block 选择 DataNode 副本位置；Client 按 128 MB 将文件切成 B1、B2、B3；Client 对每个 Block 建立 DataNode pipeline，例如 B1 写入 DN1 后转发 DN2、DN3；各 DataNode 写入本地磁盘并逐级返回确认；所有 Block 写完后 Client 通知 NameNode 完成，NameNode 更新文件元数据和 Block 位置信息。
4. 读取流程：Client 向 NameNode 请求读取 `log.dat`；NameNode 返回该文件的 Block 列表以及每个 Block 所在 DataNode 列表，如 B1 在 DN1/DN2/DN3。Client 根据就近原则或可用性选择 DataNode，直接从 DataNode 读取真实 Block 数据；真实数据不经过 NameNode。如果某个 DataNode 读取失败，Client 可切换到同一 Block 的其他副本节点。
5. DN3 故障后，B1、B2、B3 都少了一个副本：B1 还在 DN1、DN2；B2 还在 DN2、DN4；B3 还在 DN1、DN4。NameNode 通过 DataNode 周期性心跳发现 DN3 长时间无响应，将其判定为故障；再根据块汇报和副本策略发现副本数不足，选择其他 DataNode 复制缺失副本，使每个 Block 恢复到 3 个副本。
6. HDFS 适合大文件顺序读写，因为它按大 Block 存储，减少寻址和元数据开销，并通过多副本和流式读写提高吞吐。它不适合大量小文件，因为每个文件和 Block 元数据都由 NameNode 维护，会占用大量内存；也不适合低延迟随机读写，因为 HDFS 设计目标是高吞吐批处理，而不是毫秒级随机访问和频繁原地修改。

## 八、MapReduce 计算题（15 分）

1. Mapper 输入的 key/value 可以理解为输入文件中的偏移量和一行文本。`key_P` 可表示该行在输入 Split 中的偏移量或行号，`value_C` 表示该行内容，即一条“关注者 被关注者”记录。统计被关注人数时，真正用于计算的是 `value_C` 中的第二个字段“被关注者”。
2. Mapper 输出应以被关注者为 key，计数 1 为 value。

Split 1：

```text
Alice Bob   -> (Bob, 1)
Alice Carol -> (Carol, 1)
Bob Alice   -> (Alice, 1)
Bob Carol   -> (Carol, 1)
```

Split 2：

```text
Carol Alice -> (Alice, 1)
David Bob   -> (Bob, 1)
David Carol -> (Carol, 1)
Eve Alice   -> (Alice, 1)
```

3. Shuffle/Sort 后按 key 分组：

```text
(Alice, [1, 1, 1])
(Bob, [1, 1])
(Carol, [1, 1, 1])
```

4. Reducer 对每个 key 的 value 列表求和，得到每个用户被关注次数：

```text
(Alice, 3)
(Bob, 2)
(Carol, 3)
```

5. 如果统计“每个用户关注了多少人”，Mapper 输出的 key 应改为第一字段“关注者”，value 仍为 1。例如 `Alice Bob` 输出 `(Alice, 1)`，最后按关注者聚合求和。

## 九、HBase 数据建模与云存储理解（10 分）

1. HBase 是面向列族的分布式 NoSQL 数据库，适合海量数据的随机实时读写。Table 按 RowKey 排序存储，RowKey 决定数据定位和查询效率；Column Family 是物理存储和管理单位，列族下可以有多个列限定符；表按 RowKey 范围切分为多个 Region，Region 可分布在不同 RegionServer 上实现水平扩展；每个 Region 中按列族组织 Store；写入先进入 MemStore，达到阈值后刷写为 HFile/StoreFile，底层通常存放在 HDFS 上。
2. 如果主要按学生查询全部课程成绩，可以设计一张宽表 `StudentCourse`。RowKey 可用 `StudentID`，如 `S001`，便于一次读取该学生全部信息；也可用 `StudentID#Term` 支持按学期查询。列族示例：`info` 保存学生基本信息，如 `info:name`、`info:major`；`course` 保存课程成绩，如 `course:C101.name`、`course:C101.score`、`course:C101.term`、`course:C102.name`、`course:C102.score`、`course:C102.term`。
3. 一行宽表写法示例：

| RowKey | info:name | info:major | course:C101.name | course:C101.score | course:C101.term | course:C102.name | course:C102.score | course:C102.term |
| ------ | --------- | ---------- | ---------------- | ----------------- | ---------------- | ---------------- | ----------------- | ---------------- |
| S001   | Zhang     | Computer   | CloudComputing   | 92                | 2025-Fall        | Database         | 86                | 2025-Fall        |

也可以按课程拆成两行：

| RowKey    | info:name | info:major | course:name    | course:score | course:term |
| --------- | --------- | ---------- | -------------- | ------------ | ----------- |
| S001#C101 | Zhang     | Computer   | CloudComputing | 92           | 2025-Fall   |
| S001#C102 | Zhang     | Computer   | Database       | 86           | 2025-Fall   |

4. HBase 相比传统关系型数据库，弱化复杂事务和强 Join，通常不适合多表关联和复杂 SQL；但它更容易水平扩展，适合海量数据、高并发读写和按 RowKey 的快速访问。关系型数据库更强调 ACID 事务、SQL 查询和表关系约束。
